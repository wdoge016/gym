## 第3页：项目背景

**痛点分析**：

| 现状 | 问题 |
|------|------|
| 纸质登记 / Excel 管理 | 效率低、易出错、数据分散 |
| 微信/电话预约课程 | 人数统计不准、重复预约频发 |
| 器材故障无人跟进 | 报修流程不透明、维修进度不可追踪 |
| 财务手工对账 | 收入统计滞后、老板无法实时掌握经营状况 |
| 会员缺乏自助渠道 | 查课、约课、买卡都依赖到店或电话 |

**解决方案**：
打造一个打通"会员→课程→教练→器材→支付→数据"全链路的智能管理系统，一个平台解决所有环节。

## 第8页：用户认证与安全（四层防护）

**第一层 · 密码加密**：
```
注册 → BCrypt.encode(password) → 存入数据库（不可逆密文）
```
即使数据库泄露，黑客也无法还原明文密码。

**第二层 · JWT 令牌**：
```
登录成功 → 签发 JWT（有效期2小时）→ 每次请求 Header 带 Token
```
无状态认证，服务端不存 Session，水平扩展友好。

**第三层 · Token 黑名单**：
```
退出登录 → Token 加入 Redis 黑名单 → 黑名单中的 Token 后续请求直接拒绝
```
即使 Token 泄露，退出后立即失效。

**第四层 · 角色权限控制**：
```
SUPER_ADMIN → 全部权限（超管）
STORE_ADMIN → 门店管理（店长）
COACH      → 课程&考勤（教练）
STAFF      → 前台操作（前台）
```
后端通过 `@RoleCheck` 注解控制接口访问权限。

## 第9页：核心功能 — 课程预约与并发控制

**这是系统的技术核心难点。**

**场景**：动感单车课只有 20 个名额，100 人同时点"预约"。

**传统做法（有 Bug）**：
```
1. 查询 booked_count = 19
2. 判断 19 < 20，可以预约
3. UPDATE booked_count = 20
```
问题：步骤 1 和 3 之间存在时间窗口，并发时会导致超约。

**我们的方案（乐观锁）**：
```sql
UPDATE schedule
SET booked_count = booked_count + 1
WHERE id = #{id}
  AND booked_count < max_capacity
  AND status = 'AVAILABLE'
```
- 更新成功 → 预约成功
- 更新失败（影响行数=0）→ 说明名额刚好满了

**候补机制**：
名额满了自动加入候补队列 → 有人取消 → 系统自动将队首候补转为正式预约。

**爽约惩罚**：
预约未签到 → 30分钟后自动标记"爽约" → 扣 10 分信用分 → 低于 60 分禁止预约。

## 第10页：核心功能 — 会员卡与支付

**三种卡方案**：

| 卡类型 | 价格 | 有效期 |
|--------|------|--------|
| 月卡 | ¥299 | 30 天 |
| 季卡 | ¥799 | 90 天 |
| 年卡 | ¥2,599 | 365 天 |

**购买流程**：
```
选卡 → 生成订单(PENDING) → 确认支付 → 订单(PAID) → 卡片激活(ACTIVE)
```

**状态机**：
```
PENDING ──支付成功──▶ PAID ──退款──▶ REFUNDED
    │                                   
    └──超时取消──▶ CANCELLED
```

**自动汇总**：每月自动统计已支付订单金额 → 仪表盘"本月收入"指标。

## 第11页：核心功能 — AI 智能助手

**两大能力**：

**1. 智能问答**：
会员在手机端输入自然语言问题：
- "今天有什么课？"
- "怎么减脂增肌？"
- "推荐一个新手训练计划"
- "动感单车多少钱一节？"

后端调用 DeepSeek API（兼容 OpenAI 协议），将课程、教练、卡片等系统数据作为上下文注入 Prompt，让 AI 回答更精准。

**2. 课程推荐**：
根据会员历史预约记录，智能推荐可能感兴趣的课程，提升预约转化率。

**技术实现**：
```
会员端 → /api/ai/chat → AiController → DeepSeek API
                           ↓
                  注入系统上下文（课程列表、会员画像）
```

## 第16页：关键设计决策

| 决策 | 方案 | 理由 |
|------|------|------|
| 并发控制 | 乐观锁 | 冲突概率低，性能优于悲观锁，不需要锁整张表 |
| 删除策略 | 逻辑删除 | 数据可恢复，MyBatis-Plus 自动过滤，对业务透明 |
| 认证方式 | JWT 无状态 | 无需服务端 Session，便于水平扩展 |
| 接口分层 | VO / DTO 分离 | DTO 接收请求，VO 组装响应，不暴露数据库结构 |
| 异常处理 | 全局统一异常处理器 | 任何异常统一返回 `{code, msg, data}` 格式 |
| AI 集成 | 兼容 OpenAI 协议 | 可无缝切换 DeepSeek / GPT / 其他模型 |
| 前端代理 | Vite Proxy | 开发阶段解决跨域，无需后端配置 CORS |

## 第17页：总结

1. **完整的业务闭环**：会员→课程→教练→器材→支付→统计，一个系统全覆盖

6. **工程规范**：代码分层清晰、VO/DTO 分离、统一异常处理
